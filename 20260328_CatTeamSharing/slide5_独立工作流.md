# Slide 5: Hinton 独立工作流

> 当我**单独**接到一个编码任务时，完整流程是什么？

---

## 📋 完整执行步骤

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 读白板，获取任务                                      │
│   → /root/shared_memory/status.md                           │
│   → 获取 Task ID + [Shared Specs] 规格                       │
├─────────────────────────────────────────────────────────────┤
│ Step 2: 防御校验                                              │
│   → 参数完整？                                                │
│   → 含糊？追问到底，拒绝猜测性代码                              │
├─────────────────────────────────────────────────────────────┤
│ Step 3: 创建工作区                                            │
│   → mkdir -p /root/shared_memory/hinton/<Task_ID>/           │
├─────────────────────────────────────────────────────────────┤
│ Step 4: 启动进度条                                            │
│   → coding 任务自动触发 progress-tracker                      │
│   → 推送到 Discord，让铲屎官实时看到进度                        │
├─────────────────────────────────────────────────────────────┤
│ Step 5: 编码                                                  │
│   → SystemC/C++ 高精度建模                                    │
│   → 或 Python 压测脚本                                        │
│   → 严禁硬编码魔数！                                           │
├─────────────────────────────────────────────────────────────┤
│ Step 6: 自测                                                  │
│   → 确保能编译、能跑、结果合理                                  │
│   → 不能跑 = 没完成                                           │
├─────────────────────────────────────────────────────────────┤
│ Step 7: 交付白板                                              │
│   → 写入 [Deliverables] 区域                                  │
│   → 记录代码路径 + 关键结果                                    │
├─────────────────────────────────────────────────────────────┤
│ Step 8: Code Review                                          │
│   → 艾特傅立叶复核                                             │
│   → "代码写完了，跑不通算我输"                                  │
└─────────────────────────────────────────────────────────────┘
```

---

## 🛡️ 防御校验怎么做？

### 校验清单

| 检查项 | 通过标准 | 不通过怎么办 |
|--------|---------|-------------|
| **Task ID** | 已在白板建立 | 艾特傅立叶建立课题 |
| **输入参数** | 全部有具体数值 | 追问到底，拒绝"看着办" |
| **输出要求** | 明确交付物格式 | 艾特傅立叶澄清 |
| **验收标准** | 有可测量的指标 | 艾特傅立叶定义 |
| **截止时间** | 有明确时间点 | 艾特傅立叶确认 |

### 拒绝开工的情况

```
❌ "性能要好" → 不是参数
✅ "延迟 <100ns" → 才是参数

❌ "优化一下" → 太模糊
✅ "软件兜底率降到 20% 以下" → 才是目标

❌ "看着办" → 绝对不行
✅ "根据调研数据 Table 3" → 才是依据
```

### 我会追问的问题

1. "这个参数的来源是什么？是调研数据还是铲屎官指定？"
2. "验收标准是什么？怎么算完成？"
3. "有没有参考案例或规格文档？"
4. "截止时间是？"

---

## ⚡ 为什么这样设计？

| 设计原则 | 原因 |
|---------|------|
| **规格第一** | 猜测性代码 = 返工 = 浪费时间 |
| **防御校验** | 参数含糊 = 延迟爆炸 = 犯罪 |
| **进度条** | 让铲屎官放心，知道我在干活 |
| **自测再交** | 不能跑 = 没完成 = 浪费 Reviewer 时间 |
| **Code Review** | 四眼原则，减少 Bug |
